ПОВНИЙ ТЕХНІЧНИЙ ЗВІТ ПРО РЕАЛІЗАЦІЮ ЛАБОРАТОРНОЇ РОБОТИ №3
ВАРІАНТ 12: ПЛАТФОРМА ДЛЯ ЗАМОВЛЕННЯ ЇЖІ "LITTLESNACK"
================================================================================

Даний документ містить вичерпний опис архітектури та функціоналу веб-додатку, 
переведеного на фреймворк React. Проєкт реалізовано як Single Page Application (SPA)
із дотриманням усіх вимог Варіанту 12.

1. ЗАГАЛЬНА АРХІТЕКТУРА ПРОЄКТУ (React Component Model)
--------------------------------------------------------------------------------
Додаток побудований за принципом односпрямованого потоку даних (Unidirectional Data Flow).
Центральним вузлом є компонент App.js, який володіє "головним станом" (Global State).

Список ключових компонентів та їх ролі:
- App.js: Корневий компонент. Містить стан кошика (cart), смітника (trash) та замовлень (orders).
- Header.js: Навігаційна панель. Використовує Link для переходу без перезавантаження.
- Footer.js: Статичний інформаційний блок із контактами.
- MenuPage.js: Сторінка каталогу. Відповідає за рендеринг списку та фільтрацію.
- CartPage.js: Сторінка управління покупками та відновлення зі смітника.
- OrdersPage.js: Сторінка моніторингу статусу та таймерів доставки.
- CheckoutPage.js: Форма оформлення замовлення із валідацією.
- DishCard.js: Атомарний компонент картки страви.

2. ДЕТАЛЬНА РЕАЛІЗАЦІЯ ВИМОГ ВАРІАНТУ 12
--------------------------------------------------------------------------------

Вимога №1: Компоненти для картки страви, кошика та списку замовлень.
- DishCard.js: Реалізовано складну логіку відображення кнопок. Якщо товар не в кошику, 
  відображається лише "Додати". Як тільки товар додано, з'являються контролі кількості (+/-),
  а кнопка змінює статус на "Додано! (N)". Це покращує UX, фокусуючи увагу користувача.
- CartPage.js: Кошик представлений у формі таблиці з динамічними рядками. 
  Кожен рядок — це окремий об'єкт стану, що дозволяє React оновлювати лише ту частину 
  DOM, де змінилася кількість.
- OrdersPage.js: Список замовлень використовує унікальні ID та часові мітки (timestamp).
  Кожне замовлення має свій візуальний статус (Success/Rejected) та лічильник часу.

Вимога №2: Фільтрація страв за категоріями у компоненті "Меню".
- Реалізовано через керований стан activeCategory. 
- Логіка: [menuItems] -> .filter(item => item.category === activeCategory) -> .map().
- Користувач може перемикатися між категоріями "Піца", "Бургери", "Салати" тощо. 
- Також додана категорія "All" для перегляду всього асортименту. 
- Весь процес відбувається миттєво завдяки Virtual DOM та не потребує оновлення сторінки.

Вимога №3: Маршрутизація (Client-side Routing).
- Використано пакет react-router-dom версії 7+.
- Конфігурація <BrowserRouter> дозволяє використовувати URL-шляхи, які сприймаються 
  браузером як звичайні посилання, але обробляються всередині JavaScript.
- Компонент <Routes> забезпечує "розумне" перемикання сторінок: React видаляє зі сцени
  MenuPage і монтує CartPage (і навпаки), зберігаючи при цьому всі дані в пам'яті.

3. ТЕХНІЧНА РЕАЛІЗАЦІЯ БІЗНЕС-ЛОГІКИ
--------------------------------------------------------------------------------

Управління станом (Hooks):
- useState: Використовується для зберігання списку категорій, вмісту кошика, 
  видимості окремих блоків (Смітника) та значень полів форми.
- useEffect: Критично важливий хук для двох задач:
  1. Синхронізація з localStorage: при будь-якій зміні кошика дані автоматично 
     дублюються в пам'ять браузера. При перезавантаженні сторінки App.js зчитує 
     їх назад.
  2. Таймери: в OrdersPage хук useEffect запускає setInterval, який перераховує 
     залишок часу доставки (duration - (now - startTime)).

Логіка Кошика та Смітника:
- handleQtyChange: Функція високого порядку, яка оновлює кількість предмета за його ID.
- "Мінус-видалення": Якщо кількість досягає 1 і користувач натискає "-", викликається
  функція handleRemoveFromCart.
- Trash Bin System: Видалений об'єкт не зникає назавжди, а копіюється в масив trash
  із додаванням поля deletedAt. Функція handleRestoreFromTrash дозволяє повернути 
  предмет у кошик, видаливши його зі смітника.

4. ПОРІВНЯННЯ З ПОПЕРЕДНЬОЮ ВЕРСІЄЮ (Vanilla JS vs React)
--------------------------------------------------------------------------------
- Модульність: Код більше не є одним великим файлом script.js. Кожна функція та 
  компонент мають своє місце, що спрощує налагодження (debugging).
- Реактивність: У старій версії потрібно було вручну викликати renderMenu() після 
  кожної зміни. В React ми просто змінюємо стан (setCart), а фреймворк сам знає, 
  що і де треба перемалювати.
- SPA досвід: Відсутність білих спалахів між переходом по сторінках робить 
  користувацький досвід безшовним та професійним.

5. ІНСТРУКЦІЯ З РОБОТИ ТА РОЗШИРЕННЯ
--------------------------------------------------------------------------------
- Запуск: npm start (запускає dev-сервер на localhost:3000).
- Додавання страв: Редагування файлу src/data/menuData.js автоматично оновить меню 
  на сайті.
- Стилізація: Всі стилі інтегровані в src/index.css із використанням CSS Grid 
  та Flexbox, що забезпечує повну адаптивність під мобільні пристрої.

ВИСНОВОК (ПІДСУМКИ ПО ЛАБОРАТОРНІЙ РОБОТІ):
У ході виконання лабораторної роботи №3 було успішно здійснено рефакторинг веб-платформи "LittleSnack", у результаті чого додаток було переведено на компонентну архітектуру фреймворку React. Завдяки впровадженню хуків useState та useEffect вдалося реалізувати складну бізнес-логіку управління кошиком, інтерактивне меню з миттєвою фільтрацією за категоріями та систему динамічних таймерів доставки, що працюють у реальному часі. Використання бібліотеки react-router-dom дозволило створити повноцінний Single Page Application (SPA), забезпечивши безшовну навігацію між розділами без перезавантаження сторінок браузера, що значно покращило користувацький досвід (UX). Робота з localStorage гарантувала надійне збереження даних сесії, імітуючи взаємодію з базою даних на стороні клієнта. Даний проєкт наочно демонструє переваги декларативного підходу React перед імперативним JavaScript: код став більш модульним, масштабованим та легким для налагодження. В результаті отримано сучасний, швидкий та повністю адаптивний веб-додаток, який відповідає всім технічним вимогам Варіанту 12 та стандартам сучасної веб-розробки.
